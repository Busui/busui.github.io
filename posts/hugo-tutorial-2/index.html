Hugo教程系列（二）

Hugo∣ 教程∣   Hugo教程∣   2019-08-20  4291字

如果你还没有安装Hugo，请先查看Hugo教程系列（一）

看完系列一，你已经可以发布文章了。但实际上进一步操作，你还不会。这需要进一步学习，需要知道各个功能，才能掌握好Hugo。这篇文章看起来可能会吃力，但耐心看到后面系列，再回过头来看时，才能有一个比较成熟完整的概念。我作为一名开发者，也要看两遍官方文档才把Hugo搞清楚的。

Directory Structure

当你敲下hugo new site quickstart时，到底发生了什么？如果列出quickstart下的内容，很容易看到，这条命令创建了一堆文件：

.
├── archetypes
├── config.toml
├── content
├── data
├── layouts
├── static
└── themes

这堆文件结合起来就形成了某种结构，官方称之为Directory Structure(目录结构)。

Info：能够阅读英文文档和将它们翻译出来还是有一定差距的。英文中有很多东西还是需要有较好的专业素养才能翻译好的，因此，涉及到一些专有名词，为了不造成误导，我翻译原则是保留英文优先，个人翻译放后。格式如：Directory Structure(目录结构)

Hugo’s CLI(hugo的命令行接口)根据你新建的quickstart项目里面的文件，按目录结构，统一到某个虚拟的文件里，然后将这个统一的虚拟文件作为输入，生产出你的整个网站。

而从技术上来讲，hugo实际上是根据你源文件的目录和模板来作为输入，进而输出整个静态网站（下面图片的画图依据）：

按个人理解来讲，templates（模板）就是我们接下来要编写主题要做的大多数事情，模板决定了你的网站的样子。

而源文件目录的文件一般理解为资源（比如图片，文章源文件，音频……），它是你网站上的文字（图片/视频/音频等等）内容。

“整个静态网站”怎么理解呢？这其实涉及到了动态网站和静态网站的概念，动/静态网站之间的最大区别在于它们渲染出html（一般是html，这里就以之为例）页面的时机不同：

  • 动态网站：当你访问网站的某个页面时，后端代码(python,java,golang…)根据你请求的页面的需求，[可能]会查询数据库，动态生成某些信息(比如你的用户名）等等后端操作，之后才渲染成html文件，交由web服务器发给相应的浏览器。生成html页面的时机是每当有新的请求进来，web端会实时地动态地生成一个html页面。而且是每一个新的请求进来，web端都会重新生成一个html页面。（现在有缓冲技术，不一定每次都生成新的html页面文件。）

  • 静态网站：早在用户访问之前，所有页面已经存在。当有新的请求进来，web端只不过把已经存在的html页面文件返回给相应浏览器。生成html页面的时机是：用户访问之前，甚至可以说是网站发布之前。html页面文件可以是程序员自己写的，也可以交由static site generator(静态网站生成器)来生成。

如你所料，Hugo就是一个相当厉害的static site generator。所以hugo是干什么的，起码在你心中有个位置才对。

Directory Structure佛系解读

在前面，我们只是列出quickstart项目下的内容，在这一小节，我们可以大概了解一下这些文件夹的作用，如果你想真正了解每一个细节，请直接去官网查看。

themes 目录

安装主题就是安装在这个目录下。可以存放多个主题，在根目录下的config.toml文件可以配置使用哪个主题。

archetypes 目录

前一篇文章你已经知道，使用hugo new命令你可以创建新的content files(内容文件，一般是网站的文章)：

下面再来演示在quickstart目录下，使用hugo new命令新建一篇名为“这是测试hugo new命令”的文章：

~/quickstart
➜ hugo new "这是测试hugo new命令.md"
/Users/codepool/quickstart/content/这是测试hugo new命令.md created

后缀md代表这个文件格式是markdown格式。

这时候，你的content目录下会有一个名为“这是测试hugo new命令.md”的文件。（从命令的输出我们也可以看到新的文件是放在content目录下的，这是默认的），如果你点开新建的这个文件，你就会发现，里面已经存在一些东西:

---
title: "这是测试hugo New命令"
date: 2019-08-22T21:30:08+08:00
draft: true
---

默认下，根据archetypes目录下的default.md文件，hugo new命令创建出的新文件里至少包含：

  • date：创建content file的时间，格式是：2019-08-20T15:43:55+08:00；

  • title：标题，这个标题根据你创建的file名来自动生成，看它不爽，你可以随意修改。

  • draft：这是一个布尔值，默认值为true，代表[文章]是草稿，当值为false时，代表[文章]为非草稿。这个值会影响到你这篇文章是否被最终发布。

这些字段，被称为front matter。这种格式清晰简单，作用是让hugo（或其它程序）识别出你需要表达的一些信息。比如hugo通过title字段可以识别出你文章的标题。

再次提醒，hugo new命令是根据archetypes目录下的文件来决定新建文件里面的front matter的。

打开上面提到的default.md，你会发现里面也预先存了一些东西：

---
title: "{{ replace .Name "-" " " | title }}"
date: {{ .Date }}
draft: true
---

和front matter很像，但又有不同。以date举例，date对应的值不是一个具体的时间戳，而是一个表达式{{ .Date }}，当你运行hugo new命令时，{{ .Date }}会生成你输入命令摁下回车时的时间戳。default.md里面这种格式的字段被称为preconfigured front matter（预配置的front matter）。

或许你会想，这看不出front matter有多大用处啊。非也，用处可大了：

  • 这是一种统一的格式，很多软件都可以识别这种字段里面的信息。也就是说你写好一篇文章，可以拿到各种支持的平台下发布。统一意味着什么，我现在是深有体会了，真的省了很多麻烦。

  • 里面的字段是可以自定义的，而且front matter可以表达关于文章的几乎所有信息，比如作者，分类，标签……

content 目录

事实上，你写的所有文章都会放到content目录下。

如果在content目录里新建一个文件夹，那么这个文件夹是content目录里的top-level folder（顶层文件夹），每一个top-level folder会被考虑为一个content section。

例如：如果你的网站有三个主要的sections: blog, articles和tutirials，那么在你的content目录下应该有三个top-level folder：

  1. content/blogs；

  2. content/articles；

  3. content/tutorials.

hugo使用sections来决定默认的content types。比如blog下的文件，会被判定为blog type。

这有什么用呢？其中一个用处就是Hugo根据section来寻找archetypes。比如blogs session下的文件会找到archetypes/blog.md来配合hugo new命令生成新文件。如果找不到archetypes/blogs.md才会用archetypes/default.md。

config 目录

config，顾名思义，配置。你网站的所有配置，都是利用config来完成的。换句话说，通过config，可以控制hugo的行为，进而最终控制你的网站。config的形式有两种：

  1. 单一的config.toml文件。一般情况下，用单一的文件就好了。（ps:我的网站就是选择这种）

  2. 按目的或功能的不同，将各种配置按类别，分别存在不同的文件里，最后才放在config目录里。这种情况稍微复杂。如果你想进一步了解，请看configure。

配置文件支持JSON,YAML,或者TOML三种格式，本站一律采用TOML格式。如果你对TOML有门户之见，可以查看configure，改用你喜欢的。

assets 目录

如果你是linux或者mac用户，那你大概知道pipe（管道）这么一个概念。如果你不知道管道是什么，也没关系，先跳过这个概念。

简单来说，assets目录下的文件是专门给Hugo Pipes（Hugo 管道）使用的。Hugo中有专门的办法来获取assets目录下的任意资源来作为管道的输入。

默认下，assets文件夹未被创建，需要用户自己创建。

data 目录

data目录的作用有点像数据库，你可以想象一下：“在Hugo生成网站的时候，如果发现你的文章需要调用某些存储在data目录的数据，那么Hugo就会到data目录下寻找需要的数据，最后Hugo才生成最终的网站。”

一般来讲，在data目录下的数据都会呈现出某种规律，现在我们举例，假设下面的内容存在于data/testData.toml文件内：

discography = [
"1974 – Modern American Music … Period! The Criteria Sessions",
"1974 – Jaco",
"1976 - Jaco Pastorius",
"1981 - Word of Mouth",
"1981 - The Birthday Concert (released in 1995)",
"1982 - Twins I & II (released in 1999)",
"1983 - Invitation",
"1986 - Broadway Blues (released in 1998)",
"1986 - Honestly Solo Live (released in 1990)",
"1986 - Live In Italy (released in 1991)",
"1986 - Heavy'n Jazz (released in 1992)",
"1991 - Live In New York City, Volumes 1-7.",
"1999 - Rare Collection (compilation)",
"2003 - Punk Jazz: The Jaco Pastorius Anthology (compilation)",
"2007 - The Essential Jaco Pastorius (compilation)"
]

Hugo这么调用呢这些数据呢？只要使用变量.Site.Data就可以快速定位到data目录下的数据。之后就可以读取里面的数据了。如下：

{{ with $.Site.Data.testData }}
    <ul>
        {{ range .discography }}
            <li>{{ . }}</li>
        {{ end }}
    </ul>
{{ end }}

上面代码的作用是将testData.toml里面的数据以列表的形式输出。如果将testData.toml的数据增加到10000条，那么，你就能体会data目录威力的强大了。这也是我个人为什么觉得它的作用像数据库。

当然，这已经属于编程范畴了。如果你不会编程，看不懂也别慌。以后会学到。

注意：data目录下的文件格式也是支持json和yaml的。

layouts 目录

存放template的地方。前面已经提到过，hugo根据我们的templates，生成静态网站。所以你的网站看起来漂不漂亮，就看layout目录下的文件了。template文件是html格式。

模板内容十分多，是我们学习的重点。

static 目录

所有的静态内容（通常是CSS，JS，Images）都是存在在static目录下的。最大的特点就是Hugo生成静态网站时，会将整个static文件夹复制到public文件夹内。

public 目录

当你运行hugo命令，Hugo就会创建public目录。

public目录里的内容就是当你运行hugo命令时，产生的整个静态网站。换句话将，所谓的发布网站，其实就是发布public文件夹。
